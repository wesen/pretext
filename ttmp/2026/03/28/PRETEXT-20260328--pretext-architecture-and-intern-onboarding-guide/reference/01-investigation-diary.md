## Step 2: Build a ticket-local pipeline experiment and generate sample outputs

After the architecture map was clear, I added a ticket-local experiment script to make the guide less abstract. The script prints representative analysis output, prepared arrays, and line layout results for several cases that exercise important rules: soft hyphen handling, Arabic punctuation glue, URL query segmentation, CJK quote carry, and `pre-wrap` tab/hard-break behavior.

The first version failed immediately in the shell environment because there was no DOM canvas backend. Instead of forcing a browser workflow for a documentation ticket, I patched the script to install a deterministic fake `OffscreenCanvas`, matching the strategy already used by the repo’s test suite. That kept the experiment reproducible and aligned with existing repo conventions.

### What I learned
- Ticket-local experiments that touch measurement should assume no DOM canvas exists unless a browser process is explicitly involved.
- The repo’s tests already encode the right fallback pattern for this environment.

### What was tricky to build
- The trickiness was environmental, not algorithmic: the script imported the real runtime measurement layer, which assumes a browser-like canvas surface. The fix had to preserve enough behavior to make the examples meaningful without pretending to be a browser oracle.

### What warrants a second pair of eyes
- The generated numbers in `01-pipeline-samples.md` are deterministic fake-canvas outputs, not browser-accuracy outputs. That distinction is correct in the design doc, but it should stay explicit.

### What should be done in the future
- If someone wants browser-truth sample outputs inside ticket workspaces, wire a browser-backed experiment path rather than extending the fake-canvas script.

### Code review instructions
- Read the script first, then compare it to the fake `OffscreenCanvas` setup in `src/layout.test.ts`.
